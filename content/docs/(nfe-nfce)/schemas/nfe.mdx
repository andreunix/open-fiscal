Esta página é o **parser do schema da NF-e** (modelo 55): tudo o que o `leiauteNFe_v4.00.xsd` e o `tiposBasico_v4.00.xsd` fixam sobre a **forma do dado** — tipo base, tamanho, casas decimais, padrão (regex), cardinalidade (`minOccurs`/`maxOccurs`) e listas fechadas de valores — organizado em tabelas. É a *verdade de tipo e ocorrência*: o que o parser aceita **antes** de qualquer Regra de Validação.

> 🔄 Os fatos abaixo vêm do pacote `PL_010c_NT2022_002v1.30` (versão descompactada). Quando o schema diverge da Nota Técnica vigente, **o schema vence em tipo/ocorrência/enumeração** e a NT vence em regra/comportamento. Veja [Pacotes e versões](/docs/schemas/pacotes-e-versoes#divergências-conhecidas).

## Como ler estas tabelas

Cada campo do XML referencia um **tipo** (`simpleType`). O tipo carrega a validação de forma:

| Coluna | O que é |
|---|---|
| **Tipo** | nome do `simpleType` no schema (ex.: `TCnpj`, `TDec_1302`) |
| **Base** | tipo XSD subjacente (`xs:string` em quase tudo — a NF-e serializa números como texto) |
| **Tamanho** | comprimento mínimo/máximo de caracteres (`minLength`/`maxLength`) |
| **Padrão** | expressão regular (`xs:pattern`) que o valor precisa casar |
| **Ocorrência** | `minOccurs..maxOccurs` do elemento (`1..1` obrigatório único; `0..1` opcional; `1..N` lista) |

Onde houver `0|1`, `S|N` etc., o campo é uma **enumeração fechada** — só os valores listados passam. Texto livre usa `TString` (faixa `[!-ÿ]`, sem controle de conteúdo além do tamanho).

### Valores decimais (`TDec_IIDD`)

O sufixo numérico codifica a precisão: **`TDec_IIDD`** = até **II** dígitos inteiros e **DD** casas decimais. Variações de sufixo:

| Sufixo | Significado |
|---|---|
| `v` | casas decimais **variáveis** (1 a DD) |
| `Opc` | opcional — **não** aceita zero |
| `Max100` | teto de 100 (percentuais) |
| `a04` (ex.: `0302a04`) | faixa de casas decimais (de 2 a 4) |

| Tipo | Inteiros / decimais | Uso típico |
|---|---|---|
| `TDec_0104v` · `TDec_0204v` | 1–2 int / 1–4 dec | alíquotas pequenas |
| `TDec_0302a04` · `TDec_0302a04Opc` · `TDec_0302a04Max100` | 3 int / 2–4 dec | percentuais (IPI, ICMS) |
| `TDec_0302Max100` · `TDec_0304Max100` · `TDec_03v00a04Max100Opc` | 3 int / 2–4 dec, teto 100 | percentuais limitados a 100% |
| `TDec_0803v` | 8 int / 1–3 dec | quantidades |
| `TDec_1104` · `TDec_1104v` · `TDec_1104Opc` | 11 int / 4 dec | valores unitários |
| `TDec_1110v` | 11 int / 1–10 dec | quantidades de alta precisão |
| `TDec_1203` | 12 int / 3 dec | quantidade tributável |
| `TDec_1204` · `TDec_1204v` · `TDec_1204Opc` · `TDec_1204temperatura` | 12 int / 4 dec | valores unitários grandes |
| `TDec_1302` · `TDec_1302Opc` | 13 int / 2 dec | **valores monetários** (vBC, vICMS, vNF…) |

> Os `TDec_*` **não** têm `fractionDigits` no schema — a precisão é imposta pela própria regex (ex.: `TDec_1302` = `0|0\.[0-9]{2}|[1-9]{1}[0-9]{0,12}(\.[0-9]{2})?`). O separador decimal é o **ponto**, sem separador de milhar e sem zeros à esquerda.

## Campos que mudam — CNPJ alfanumérico

A **NT 2026.004** torna o CNPJ alfanumérico. Os tipos de documento e a **chave de acesso** ganham letras nas 12 primeiras posições do CNPJ (os 2 últimos dígitos verificadores seguem numéricos). O schema novo está no pacote `PL_010d_NT2026.004`.

| Tipo | Padrão atual (`PL_010c`) | Padrão novo (`PL_010d` / NT 2026.004) |
|---|---|---|
| `TCnpj` | `[0-9]{14}` | `[0-9A-Z]{12}[0-9]{2}` |
| `TCnpjOpc` | `[0-9]{0}\|[0-9]{14}` | `[0-9]{0}\|[0-9A-Z]{12}[0-9]{2}` |
| `TCnpjVar` | `[0-9]{3,14}` | `[0-9A-Z]{12}[0-9]{2}` *(passa a 14 fixo)* |
| `TChNFe` | `[0-9]{44}` | `[0-9]{6}[0-9A-Z]{12}[0-9]{26}` |

### Quando entra — homologação e produção

| Marco | Data |
|---|---|
| **Homologação** (ambiente de teste) | **01/06/2026** (disponibilização até 15/06/2026) |
| **Produção** (passa a valer) | **01/07/2026** |

> Até **30/06/2026** o parser de produção exige CNPJ/chave **numéricos** (`[0-9]{14}` / `[0-9]{44}`). A partir de **01/07/2026** passa a aceitar o padrão alfanumérico. Quem testa contra homologação deve adotar o novo padrão já a partir de **01/06/2026**. Implemente a troca por **feature flag de data**, não por substituição direta.
>
> **Divergência conhecida:** o `consCad_v2.00` baixado fixa `TCnpjVar` em 14 posições, enquanto a NT 2026.004 prevê 3–14 caracteres na *consulta cadastro*. Veja [Consulta cadastro](/docs/emissao-e-comunicacao/consulta-cadastro) e [Chave de acesso](/docs/fundamentos/chave-de-acesso).
